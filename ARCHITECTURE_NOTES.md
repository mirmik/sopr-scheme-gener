# Анализ состояния проекта

## Общая оценка

Проект выглядит как рабочий GUI-прототип, который долго развивался добавлением новых возможностей поверх существующего кода. Он небольшой и синтаксически собирается, но как поддерживаемая кодовая база находится в плохом состоянии: много глобального состояния, слабая воспроизводимость окружения, нет тестов, а архитектура рисования не выделена в самостоятельную систему.

Ориентировочная оценка технического долга: 7/10. Если учитывать открытие внешних файлов схем через `pickle`, риск ближе к 8/10.

## Основные проблемы

### Воспроизводимость

В проекте нет явного описания зависимостей: отсутствуют `requirements.txt`, `pyproject.toml`, `setup.py` или аналогичный файл. В текущем окружении `PyQt5` не установлен, поэтому приложение нельзя запустить без ручного восстановления зависимостей.

Сборочный `.spec` файл содержит захардкоженный путь `C:\sopr-scheme-gener`, что делает сборку непереносимой.

### Безопасность сериализации

Сохранение и загрузка схем построены на `pickle`:

- `sopr-sheme-gener.py`
- `common.py`
- `taskconf_menu.py`

Это опасно, если `.dat` файлы могут приходить извне: загрузка pickle-файла потенциально позволяет выполнить произвольный код. Для локального личного использования это терпимый исторический компромисс, но для обмена схемами такой формат лучше заменить.

### Глобальное состояние

Модуль `common.py` используется как глобальный контейнер состояния:

- `APP`
- `CONFVIEW`
- `SCHEMETYPE`
- `HSPLITTER`
- `PAINT_CONTAINER`

Из-за этого порядок инициализации становится критичным, отдельные части системы трудно тестировать, а зависимости между UI, моделью и рендером становятся неявными.

### Точка входа

В `sopr-sheme-gener.py` приложение запускается прямо на верхнем уровне файла: парсятся аргументы, создается `QApplication`, окно показывается, запускается event loop. Нет защитного блока `if __name__ == "__main__"`.

Это мешает импортировать модуль в тестах или переиспользовать его части.

### Обработка ошибок

Часть ошибок подавляется без сообщения. Например, редактирование таблицы в `tablewidget.py` обернуто в голый `except: pass`. В результате некорректный пользовательский ввод или баги в модели могут не проявляться явно, но оставлять приложение в неправильном состоянии.

Есть и более конкретные проблемы: при загрузке, если связанный `.dat` не найден, показывается сообщение об ошибке, но выполнение может продолжиться с `None`.

## Подход к рисованию

Рисование выглядит несистемно. Архитектура скорее процедурная и накопительная: каждый тип задачи сам вычисляет координаты, размеры, подписи, стрелки, подсветки и сразу вызывает `QPainter` или функции из `paintool.py`.

### Что не так

#### Нет промежуточной модели сцены

Код не строит декларативное описание схемы в виде объектов вроде:

- `Beam`
- `Force`
- `Moment`
- `DimensionLine`
- `Support`
- `TextLabel`
- `SectionShape`

Вместо этого он сразу рисует пиксели. Поэтому поведение сложно тестировать без запуска GUI и фактического рендера.

#### Расчет геометрии смешан с отрисовкой

Внутри `paintEventImplementation()` одновременно:

- читается состояние задачи;
- вычисляются масштабы и координаты;
- выбираются варианты отображения;
- рисуются линии, стрелки, текст и подсветки;
- учитываются пользовательские настройки.

Такой код сложно менять локально: изменение одного элемента может повлиять на раскладку других элементов.

#### Много магических чисел

В коде много локальных констант вроде `10`, `20`, `40`, `50`, `3.2`, ручных смещений текста и стрелок. Они не собраны в систему правил компоновки, поэтому поведение рендера приходится понимать по месту.

#### `paintool.py` не является полноценным движком

`paintool.py` содержит много процедурных примитивов для рисования стрелок, линий, текста и специальных обозначений. Это полезная библиотека низкоуровневых функций, но она не задает архитектуру сцены и не отделяет модель от визуализации.

#### Новые типы задач добавляются копированием паттерна

Файлы в `tasks/` выглядят как отдельные специализированные реализации. При добавлении нового типа задачи создается новый набор таблиц, состояния и процедур рисования. Общего расширяемого рендер-пайплайна почти нет.

#### Интерактивность привязана к рендеру

Hover, подсветка, перетаскивание меток и связь с таблицами завязаны на внутренние поля виджетов рисования. Нет общей системы hit-testing по объектам сцены.

## Возможное направление рефакторинга

Не стоит сразу переписывать весь проект. Более безопасный путь:

1. Зафиксировать окружение и зависимости.
2. Вынести запуск приложения в `main()`.
3. Добавить минимальные smoke-тесты на импорт, создание базовой схемы, сериализацию и рендер в изображение.
4. Ввести промежуточный слой сцены.
5. Перевести один тип задачи, например `task0`, на генерацию сцены вместо прямого рисования.
6. Постепенно переносить остальные типы задач.

## Предлагаемая модель сцены

Минимальная идея:

```text
Task model -> Layout builder -> Scene objects -> Renderer -> QPainter
```

Где:

- `Task model` хранит предметные данные задачи;
- `Layout builder` вычисляет координаты и размеры;
- `Scene objects` описывают, что должно быть нарисовано;
- `Renderer` умеет рисовать эти объекты через `QPainter`.

Примерные объекты сцены:

- `Line`
- `Arrow`
- `Text`
- `Dimension`
- `Support`
- `DistributedLoad`
- `Torque`
- `Section`
- `Highlight`

Это позволит:

- тестировать геометрию без GUI;
- переиспользовать элементы между типами задач;
- отделить данные от визуального представления;
- сделать hit-testing системным;
- уменьшить количество ручных поправок в коде задач.

## Приоритеты

Самые полезные первые шаги:

1. Описать зависимости проекта.
2. Убрать запуск приложения с верхнего уровня `sopr-sheme-gener.py`.
3. Перестать использовать `pickle` для новых файлов схем.
4. Добавить минимальные тесты.
5. Начать выделение сцены с одного самого простого типа задачи.

Главный вывод: проект не безнадежный, но сейчас он держится на процедурном коде и неявных связях. Рефакторинг лучше начинать не с эстетики, а с создания тонкой архитектурной прослойки вокруг рисования.
