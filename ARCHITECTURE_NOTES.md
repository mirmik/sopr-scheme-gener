# Анализ состояния проекта

## Общая оценка

Проект выглядит как рабочий GUI-прототип, который долго развивался добавлением новых возможностей поверх существующего кода. Он синтаксически собирается и запускается, но как поддерживаемая кодовая база пока находится в плохом состоянии: много глобального состояния, лишь минимальный набор тестов, а архитектура рисования не выделена в самостоятельную систему.

В июле 2026 года создан внешний baseline: пакет `sopr_scheme_gener`, нормальная
точка входа, `pyproject.toml`, реестр типов со стабильными ID, TCP development
API, клиент `soprctl`, `AppContext` и `DocumentController`. Все публикации
старых `common`-глобалов сосредоточены в `LegacyAdapter`. Этот слой намеренно
не меняет legacy-рендереры и модели задач.

Ориентировочная оценка технического долга: 7/10. Если учитывать открытие внешних файлов схем через `pickle`, риск ближе к 8/10.

## Основные проблемы

### Воспроизводимость

Runtime-зависимости теперь описаны в `pyproject.toml`, а wheel собирается
локально. Нужны lock/constraints для полностью повторяемого development
окружения и проверка установки в чистом окружении.

Исторические `.spec`-файлы по-прежнему содержат захардкоженный путь
`C:\sopr-scheme-gener` и некорректные entry scripts. Windows-сборка пока не
воспроизводима.

### Безопасность сериализации

Новые документы используют версионированный `.sopr.json`: envelope и payload
конкретного типа имеют отдельные версии, модельные классы выбираются только из
allowlist по stable task ID. Имена Python-модулей из файла не импортируются.
Open/Save document и Export image разделены.

Исторический pickle-код остаётся только для явно выбранного пользователем
импорта доверенного `.dat`. Автоматический поиск pickle-sidecar по хешу
изображения удалён. После завершения переходного периода legacy codec можно
будет удалить вместе с `common.SchemeType.serialize/deserialize`.

### Глобальное состояние

Модуль `common.py` используется как глобальный контейнер состояния:

- `APP`
- `CONFVIEW`
- `SCHEMETYPE`
- `HSPLITTER`
- `PAINT_CONTAINER`

Из-за этого порядок инициализации становится критичным, отдельные части системы трудно тестировать, а зависимости между UI, моделью и рендером становятся неявными.

### Точка входа

Точка входа перенесена в `sopr_scheme_gener.app:main`; поддерживаются
`python -m sopr_scheme_gener` и установленная команда
`sopr-scheme-gener`. Исторический `sopr-sheme-gener.py` оставлен тонким
совместимым launcher'ом. Application shell и development API работают через
`AppContext`; прямой доступ нового package-кода к `common`-глобалам запрещён
архитектурным тестом.

### Автоматизация и тесты

Добавлены unit/smoke-тесты, внешний baseline из 22 сценариев и TCP development
API. `soprctl` умеет получать состояние приложения, перечислять и выбирать типы
задач, читать Qt-объекты, снимать PNG, структурно читать и атомарно изменять
активный документ. Одним запросом можно выполнить сценарий выбора/изменения,
дождаться обработки событий Qt и получить снимок.

Журналы событий и ошибок имеют монотонные курсоры. При активном development API
исключения рендера попадают в журнал вместо блокирующего диалога. Произвольный
Python внутри GUI-процесса остаётся отдельной opt-in возможностью.

### Обработка ошибок

Часть ошибок подавляется без сообщения. Например, редактирование таблицы в `tablewidget.py` обернуто в голый `except: pass`. В результате некорректный пользовательский ввод или баги в модели могут не проявляться явно, но оставлять приложение в неправильном состоянии.

Есть и более конкретные проблемы: при загрузке, если связанный `.dat` не найден, показывается сообщение об ошибке, но выполнение может продолжиться с `None`.

## Подход к рисованию

Рисование выглядит несистемно. Архитектура скорее процедурная и накопительная: каждый тип задачи сам вычисляет координаты, размеры, подписи, стрелки, подсветки и сразу вызывает `QPainter` или функции из `paintool.py`.

### Что не так

#### Нет независимой промежуточной модели сцены

Код не строит декларативное описание схемы в виде объектов вроде:

- `Beam`
- `Force`
- `Moment`
- `DimensionLine`
- `Support`
- `TextLabel`
- `SectionShape`

Большинство типов сразу рисует пиксели. `vali` и `balki3d` частично строят
`QGraphicsScene` и используют объекты из `items/`, но делают это внутри
`paintEvent`. Это Qt-зависимое представление, а не тестируемая декларативная
модель сцены. `cube` уже перенесён на независимый `StressCubeLayoutBuilder`;
`QGraphicsScene` для него остался только внутри Qt-адаптера растровой
совместимости.

#### Расчет геометрии смешан с отрисовкой

Внутри `paintEventImplementation()` одновременно:

- читается состояние задачи;
- вычисляются масштабы и координаты;
- выбираются варианты отображения;
- рисуются линии, стрелки, текст и подсветки;
- учитываются пользовательские настройки.

Такой код сложно менять локально: изменение одного элемента может повлиять на раскладку других элементов.

#### Много магических чисел

В коде много локальных констант вроде `10`, `20`, `40`, `50`, `3.2`, ручных смещений текста и стрелок. Они не собраны в систему правил компоновки, поэтому поведение рендера приходится понимать по месту.

#### `paintool.py` не является полноценным движком

`paintool.py` содержит много процедурных примитивов для рисования стрелок, линий, текста и специальных обозначений. Это полезная библиотека низкоуровневых функций, но она не задает архитектуру сцены и не отделяет модель от визуализации.

#### Новые типы задач добавляются копированием паттерна

Файлы в `tasks/` выглядят как отдельные специализированные реализации. При добавлении нового типа задачи создается новый набор таблиц, состояния и процедур рисования. Общего расширяемого рендер-пайплайна почти нет.

#### Интерактивность привязана к рендеру

У ещё не перенесённых типов hover, подсветка, перетаскивание меток и связь с
таблицами завязаны на внутренние поля виджетов рисования. Для `beams` и
`stress-cube` это уже заменено общим `SceneIndex` по stable `object_id`.

## Принятое направление рефакторинга рендера

Архитектурный совет принял Qt-независимую неизменяемую Scene-модель как
контракт между task/layout и renderer. Полное решение, альтернативы и риски
зафиксированы в
[`docs/architecture-council/2026-07-23-scene-api.md`](docs/architecture-council/2026-07-23-scene-api.md).

Миграция выполняется небольшими проверяемыми этапами:

1. Минимальное Scene core и `QtPainterRenderer` — выполнено.
2. Первый пилот на типичном процедурном 2D-рендере `beams` — выполнено:
   `BeamLayoutBuilder` и layout поперечных сечений не зависят от Qt,
   предметный legacy `QPainter`/`paintool` path удалён, default и trusted
   legacy baseline сохранены.
3. Проверка общности на `stress-cube` — выполнено: геометрия проекции, кубов,
   напряжений, моментов, подписей и viewport вынесена из виджета; все шесть
   default/trusted raster baseline сохранены побитово.
4. Общий hit-testing по stable `object_id` — выполнено: `SceneIndex`
   вычисляет bounds, вложенные offsets и z-order без Qt, а
   `QtSceneInteraction` переводит события виджета в координаты сцены.
5. Первый тип следующей волны, `plate`, — выполнено: контуры, штриховки,
   размеры, нагрузки, моменты, опоры и draggable labels используют общий
   Scene pipeline; default baseline сохранён побитово.
6. Родственные `rod-system-1` и `rod-system-2` — выполнено отдельными
   builder'ами: линейная и графовая модели не смешаны, а интерактивное
   построение ветвей использует `SceneIndex`.
7. `frames` — выполнено: геометрия стержней, опоры, нагрузки, силы, моменты,
   подписи и поперечное сечение вынесены в чистый `FramesLayoutBuilder`;
   grid hover, preview и создание стержня работают по семантическим
   `grid-node`, default baseline сохранён побитово.
8. `axial-torsion` — выполнено: общий `AxialTorsionLayoutBuilder` покрывает
   растяжение/сжатие и оба режима кручения, сохраняя их различную силовую
   символику; default baseline сохранён побитово.
9. `oblique-bending` — выполнено: проекция стержня, консоль, размеры,
   сосредоточенные и распределённые силы, моменты и пространственные опоры
   вынесены в `ObliqueBendingLayoutBuilder`; default и trusted baseline
   сохранены побитово.
10. `eccentric-bending` — выполнено: четыре формы сечения, пространственное
    представление консоли, размеры и силы в восьми опорных точках вынесены в
    `EccentricBendingLayoutBuilder`; default baseline сохранён побитово.
11. `shafts-pipes` — выполнено: сплошные валы, полые трубы, центральная
    секция, камера, торцевые разрезы, нагрузки, моменты и размеры вынесены в
    `ShaftsPipesLayoutBuilder`; default и четыре trusted baseline сохранены
    побитово.
12. Далее последний предметный 3D-тип `spatial-beams` (`balki3d`).

Windows-сборка остаётся финальным ручным gate и не блокирует эту работу.

## Контракт Scene

Минимальная идея:

```text
Task model -> Layout builder -> Scene objects -> Renderer -> QPainter
                                      |
                                      +-> SceneIndex -> interaction adapter
```

Где:

- `Task model` хранит предметные данные задачи;
- `Layout builder` вычисляет координаты и размеры;
- `Scene objects` описывают, что должно быть нарисовано;
- `Renderer` умеет рисовать эти объекты через `QPainter`.

Начальный набор объектов сцены:

- `Line`
- `Arrow`
- `Text`
- `Polyline`
- `Polygon`
- `Group`

Пилот `beams` подтвердил необходимость также общих `Rectangle`, `Ellipse`,
`Arc`, patterned `Fill`, dash-dot `Stroke` и baseline anchors текста. Пилот
`stress-cube` не потребовал новых примитивов: кубы, оси, напряжения и моменты
выразились теми же `Line`, `Polygon`, `Text` и `Group`. При этом силы, опоры,
размеры, сечения, напряжения и моменты остались семантическими `Group` с
устойчивыми `object_id`, а не стали предметными mutable-примитивами Scene core.

Перенос `plate` подтвердил переиспользование того же набора для вложенных
контуров, пар сил, распределённых нагрузок и опор. В низкоуровневый контракт
добавлены только общие растровые свойства: forward-diagonal `Fill` и
`Polygon.convex`, необходимый для точного Qt `drawConvexPolygon`.

`rod-system-1` также переведён на отдельный чистый layout builder. Его Scene
покрывает секционные размеры и подписи, положительные и отрицательные стержни,
зазоры, локальные и парные силы, промежуточные и концевые опоры. Скрытый
legacy-побочный эффект перезапуска `QPainter` выражен явным необязательным
`Group.antialias`: это сохраняет побитовый baseline, не связывая предметную
геометрию с Qt.

Родственный `rod-system-2` сохранён отдельным builder: его модель — граф
ветвящихся стержней с углами и силами, а не вариант вертикальной системы.
Узлы графа представлены прозрачными семантическими объектами Scene; hover,
выбор начала новой ветви и preview используют общий `SceneIndex`. Таким
образом, интерактивное построение больше не вычисляет собственную карту
попаданий поверх процедурного рендера.

`frames` переиспользует общий builder поперечных сечений из `beams`, не
дублируя его Qt-отрисовку. Семантические группы стержней, опор,
распределённых нагрузок, локальных сил и моментов имеют устойчивые
`object_id`. Интерактивная координатная сетка также входит в Scene: её
прозрачные области попадания индексируются вместе с видимой геометрией, а
виджет только преобразует результат hit-test в новую запись модели.

`axial-torsion` использует один геометрический каркас секций, оси, размеров,
заделок и подписей, но отдельные ветви layout для осевых сил и крутящих
моментов. Это сохраняет три пользовательских подтипа без возврата к
монолитному `paintEvent`. В ходе переноса устранён старый Qt runtime-crash
размерных подписей при дробной геометрии в режиме кручения по диаметрам.

`oblique-bending` ввёл общий Qt-независимый `Projection3D`: он сохраняет
историческую математику ортографической, 45-градусной и аксонометрической
проекций, но возвращает Scene `Point`. Проектор не знает о painter или
предметных объектах; силы, моменты и опоры остаются семантическими группами
конкретного layout.

`eccentric-bending` подтвердил повторное использование `Projection3D` для
другой модели: прямоугольного, ромбического, круглого и трубчатого сечений с
пространственными силами. Старые отладочные `print` из пути рисования удалены;
политики размещения силовых подписей теперь явно представлены metadata групп.

`shafts-pipes` использует Scene как предметный контракт даже для исторически
сложной композиции `QGraphicsItem`. Особые правила границ, текста и
одномасштабного растрового вывода локализованы в `QtGraphicsSceneRenderer`;
layout и тесты геометрии Qt не импортируют. Тот же compatibility renderer
сохраняет побитовое наследие `stress-cube`.

Интерактивность использует тот же неизменяемый снимок Scene:

- `SceneIndex` вычисляет консервативные bounds примитивов и семантических
  групп, учитывая вложенные `Group.offset`;
- `hit_test()` фильтрует кандидатов по metadata и выбирает верхний объект по
  painter z-order, при равенстве — наиболее глубокий;
- `ViewportMapping` задаёт direct и aspect-fit преобразования координат;
- `QtSceneInteraction` остаётся тонким адаптером `QPointF` и не переносит Qt
  в Scene core.

`beams`, `stress-cube`, `plate`, `rod-system-2`, `frames` и
`axial-torsion` больше не строят локальные `TextItem`/hover maps для
перенесённых подписей; hover и drag обращаются к одному API по `object_id`.

Предметные `Dimension`, `Support`, `DistributedLoad`, `Torque`, `Section` и
`Highlight` добавляются только при подтверждённом повторном использовании.

Это позволит:

- тестировать геометрию без GUI;
- переиспользовать элементы между типами задач;
- отделить данные от визуального представления;
- сделать hit-testing системным;
- уменьшить количество ручных поправок в коде задач.

## Приоритеты

Самые полезные первые шаги:

1. Перенести остальные процедурные 2D-типы.
2. После этого перенести `vali` и `balki3d`, используя опыт compatibility
   adapter из `stress-cube`.
3. Восстановить воспроизводимую Windows-сборку в самом конце.

Главный вывод: проект не безнадежный, но сейчас он держится на процедурном коде и неявных связях. Рефакторинг лучше начинать не с эстетики, а с создания тонкой архитектурной прослойки вокруг рисования.
