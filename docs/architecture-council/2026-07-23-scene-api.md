# Контракт независимой Scene-модели

Дата: 2026-07-23  
Статус: Accepted

## Контекст

Одиннадцать типов задач исторически совмещают чтение предметной модели,
вычисление раскладки, изменение размера canvas, hit-testing и непосредственные
вызовы `QPainter`/`paintool` внутри `paintEvent`. Несколько 3D-задач локально
используют `QGraphicsScene`, но не разделяют общий контракт сцены.

После появления render baseline, `AppContext`, структурного development API и
версионированного документа рендер можно переносить постепенно, проверяя
визуальную эквивалентность каждого шага.

## Принятое решение

Ввести собственную Qt-независимую Scene-модель как единственный новый контракт
между layout-кодом задачи и отображением:

```text
Task document + settings
          |
          v
LayoutBuilder.build(..., TextMetrics)
          |
          v
Scene(viewport, objects)
          |
          v
QtPainterRenderer
```

Основные правила:

1. Core scene/value objects неизменяемы и не импортируют PyQt.
2. Первый этап использует пиксельные координаты, чтобы сохранить текущий
   визуальный результат.
3. Layout возвращает `viewport`/`bounds`; изменение размеров QWidget не
   является побочным эффектом рисования.
4. Измерение текста задаётся интерфейсом `TextMetrics`. Первая реализация
   использует Qt, а unit tests могут применять детерминированную подмену.
5. Стили являются значениями Scene, а не состоянием глобальных pen/brush.
6. Интерактивные семантические элементы получают стабильный `object_id`.
7. `QPainter` и `QGraphicsScene` допустимы только внутри UI/renderer adapters,
   но не в task model, Scene core или layout contract.
8. Scene строится заново из документа; она не становится вторым изменяемым
   источником истины.

Минимальное ядро включает геометрию и стили, `Line`, `Polyline`, `Polygon`,
`Text`, `Arrow`, `Group`, Scene traversal и Qt painter adapter. Специализированные
инженерные примитивы добавляются только после подтверждения повторного
использования в пилотах.

## Обоснование

- Независимый layout можно тестировать без QApplication и raster backend.
- Один Scene можно в дальнейшем отрисовать в Qt, экспортировать или
  диагностировать через development API.
- Стабильные ID дают общий путь к hit-testing вместо локальных hover maps.
- Qt остаётся практическим backend для точного сохранения текущих шрифтов и
  baseline, не проникая в предметный контракт.
- Пиксельные координаты уменьшают риск первой миграции; переход к иным единицам
  можно обсуждать после двух пилотов.

## Рассмотренные альтернативы

### Сохранить непосредственный QPainter

Самый дешёвый локальный рефакторинг, но layout, rendering и interaction
остаются связанными. Не создаёт тестируемой границы и почти не помогает
последующей миграции.

### Использовать QGraphicsScene как общий формат

Упрощает Qt hit-testing и transforms, однако делает layout и тесты зависимыми
от Qt, а экспорт и альтернативные backend — от структуры виджетов. Допустим
как adapter, но отвергнут как предметная Scene-модель.

### Сразу ввести физические/нормализованные единицы

Архитектурно привлекательно, но одновременно меняет layout semantics и
растеризацию всех существующих задач. Отложено до стабилизации двух пилотов.

## Последствия и риски

- Временно будут сосуществовать новый и legacy render paths.
- Совпадение текста зависит от корректности Qt `TextMetrics` и правил baseline.
- Слишком ранняя библиотека предметных примитивов может закрепить неверные
  абстракции; сначала используются низкоуровневые primitives.
- Общий hit-testing откладывается до опыта `beams` и `stress-cube`.
- Нельзя переносить все типы одним большим изменением: каждый пилот обязан
  сохранять соответствующие default и trusted legacy baseline.
